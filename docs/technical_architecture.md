# SeeSea 技术架构说明

## 1. 总体架构

SeeSea 采用“用户交互层 + 业务编排层 + AI 决策层 + 空间智能层 + 数据层”的分层结构。

```text
用户交互层
  潜水员端 / 管理员端 / 手机网页 / 主机端展示

业务编排层
  登录鉴权 / 日历任务 / 案例管理 / 审核流 / 任务派发 / 报告发布

AI 决策层
  图像增强 / 多模型识别 / OCR / 风险融合 / 证据链 / 多 Agent 报告

空间智能层
  高德地图 / PostGIS / 区域查询 / 漂流轨迹 / OpenDrift 接口

数据层
  用户 / 潜点 / 作业批次 / 案例 / 垃圾身份证 / 证据 / 任务 / 报告
```

## 2. 前端设计

- 响应式 Web。
- 同一套界面支持手机端、桌面端和主机端。
- 手机端底部导航，便于潜水员现场操作。
- 管理员端采用 CRM 式后台资源管理。
- 地图支持拖动、缩放和点击选点。
- 关键图片以原图、AI 增强图、AI 识别图三联方式展示。

## 3. 后端设计

- 使用 FastAPI 作为 API 服务框架。
- REST API 提供登录、案例、垃圾、任务、漂流分析和报告接口。
- 启动时自动初始化数据库结构和演示数据。
- 登录支持账号密码，演示账号默认密码为 `seesea123`。

## 4. 数据库设计

系统从 SQLite 升级为 PostgreSQL + PostGIS。

主要数据对象：

- `users`：用户、角色和密码哈希。
- `dive_sites`：潜点和空间坐标。
- `dive_sessions`：潜水员作业批次。
- `cases`：每张图片对应的案例。
- `debris_items`：垃圾身份证。
- `evidence_records`：AI 证据链。
- `weighing_records`：称重记录。
- `volunteer_feedback`：志愿者心声。
- `tasks`：治理任务，包含自定义地图点。
- `drift_runs`：漂流溯源分析结果。
- `reports`：多智能体治理报告。
- `audit_logs`：人工复核和审计记录。

PostGIS 用于：

- 垃圾点位存储。
- 任务点位存储。
- 潜点空间数据。
- 区域半径查询。
- 后续漂流轨迹和空间统计扩展。

## 5. AI 管线

系统采用可替换的 AI 管线：

1. 水下图像增强：预留 FUnIE-GAN 或同类模型接口。
2. 目标检测：预留 YOLO 系列模型接口。
3. OCR：识别包装文字和标签线索。
4. VLM 大模型：结合图片和文本进行语义识别。
5. 融合引擎：整合类别、材质、数量、风险、来源活动。
6. 人工复核：管理员和潜水员修正结果后写入证据链。

## 6. AI 证据链

每条 AI 判断都保存：

- 证据类型。
- 来源名称。
- 模型版本。
- 输入引用。
- 置信度。
- 判断解释。
- 人工审核状态。
- 创建时间。

这样可以避免只有一个不可解释识别结果，提升公益数据可信度。

## 7. 漂流溯源技术

漂流沙盘采用 OpenDrift / Lagrangian particle tracking 的技术思路。

当前设计支持：

- 单个垃圾点分析。
- 区域垃圾聚合分析。
- 反向溯源。
- 未来扩散。
- 不确定性区间。
- 区域内垃圾样本展示。

未来可接入：

- 海流数据。
- 潮汐数据。
- 风速风向。
- 浪高。
- 海岸线地形。
- 河口、港口、航道和渔区数据。

## 8. 多智能体报告

报告中心不是简单文本生成，而是多 Agent 协同推理：

- 视觉分析 Agent。
- 证据链审计 Agent。
- 溯源推理 Agent。
- 安全评估 Agent。
- 行动规划 Agent。
- 公益影响 Agent。
- 志愿者反馈 Agent。
- 总控调度 Agent。

报告内容包括：

- KPI 快照。
- 垃圾身份证统计。
- 材质分布。
- 任务联动。
- 漂流沙盘引用。
- 志愿者反馈。
- 行动建议。

## 9. 技术效果

- 现场采集数据结构化。
- 垃圾识别结果可追溯。
- 空间分布和漂流分析可视化。
- 管理员审核和任务派发闭环化。
- 公益成果可报告化和课件化。
- 系统可从单机构工具扩展为区域海洋垃圾治理数字底座。
