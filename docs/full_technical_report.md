# SeeSea 完整技术报告

## 1. 项目概述

SeeSea 是一套面向海洋垃圾治理行动的 AI 协同系统，围绕“水下图像增强与还原”“垃圾身份证识别与核对”“双端协作管理”“志愿者反馈语义分析”“污染源与扩散路径推演”等需求构建。

项目目标不是做单点识别工具，而是搭建一条从前线采集到后台研判、从单案例分析到项目级治理决策的完整数据链路。

## 2. 设计目标

本系统围绕赛事评审指标进行设计：

### 2.1 技术实现难度

- 多模态输入：图像、OCR 文本、人工表单、志愿者反馈
- 多阶段处理：增强、识别、核对、语义分析、轨迹模拟、报告生成
- 双角色协同：潜水员端与管理员端
- 模型混合架构：传统视觉、本地 ONNX、云端视觉模型、云端文本模型

### 2.2 创新性

- 提出“垃圾身份证”概念
- 将“图像”转为“结构化证据链”
- 将识别、来源推断、表单核对、扩散路径连接为闭环

### 2.3 功能完整度

- 覆盖采集、识别、审核、导出、展示全流程
- 支持移动端拍照、地图选点、管理员审核、报告导出

### 2.4 可实施性

- 采用轻量技术栈
- 支持本地降级运行
- 公益机构可在单机环境中部署

### 2.5 可持续性

- 数据可长期积累
- 模型接口可替换
- 可向更多潜点、更多机构和更多样本扩展

## 3. 总体架构

## 3.1 架构分层

系统采用四层结构：

### A. 交互层

- 管理员端 Web 页面
- 潜水员端移动友好 Web 页面

### B. 业务层

- 用户登录与角色鉴权
- 案例创建、删除、审核、导出
- 工作流状态管理

### C. AI 服务层

- 图像增强模块
- 垃圾识别模块
- OCR 模块
- 表单核对模块
- 语义分析模块
- LLM 报告模块
- 扩散路径模拟模块

### D. 数据层

- SQLite 数据库
- 图片与标注文件
- JSON 结构化结果

## 3.2 技术栈

### 后端框架

- FastAPI
- SQLAlchemy
- Starlette SessionMiddleware

### 数据库

- SQLite
- 启用 WAL 模式和 busy timeout

### 前端

- 原生 HTML
- 原生 CSS
- 原生 JavaScript
- Leaflet 地图组件

### 图像处理

- OpenCV
- Pillow

### AI 与模型接口

- 本地规则识别
- 本地 ONNX 模型接口
- SiliconFlow 视觉模型接口

### 外部模型

- 视觉模型：`THUDM/GLM-4.1V-9B-Thinking`

## 4. 核心模块与实现方式

## 4.1 登录与权限控制

### 目标

保证管理员与潜水员的权限隔离，并支持同浏览器双角色操作。

### 实现

- 登录接口签发 token
- 前端本地保存角色独立 token
- API 通过 Bearer Token 认证
- 管理员和潜水员分别拥有独立访问范围

### 价值

- 避免 session 相互覆盖
- 更适合双端演示和真实运营

## 4.2 潜水员端采集

### 目标

让潜水员能在手机端低门槛地完成现场数据采集。

### 实现

- `拍照上传` 按钮动态触发 `capture=environment`
- `选择图片` 支持相册上传
- 上传前提供缩略图预览
- 表单支持人工条目录入和备注补充

### 技术点

- HTML file input
- 动态 capture 策略
- 原生 JavaScript 文件预览

## 4.3 定位与地图选点

### 问题

手机浏览器在 HTTP 页面下经常拒绝系统定位。

### 实现

- 优先调用 `navigator.geolocation`
- 若失败，则自动切换到 Leaflet 地图选点
- 用户在地图点击后自动回写经纬度

### 价值

- 保证移动端演示稳定
- 提升现场可用性

## 4.4 图像增强

### 目标

提升原始水下图像可读性，为识别和展示提供高质量输入。

### 实现思路

- 亮度与对比度增强
- 色偏纠正
- 局部清晰度提升

### 技术

- OpenCV 图像处理

### 输出

- 原图
- 增强图

## 4.5 垃圾识别

### 目标

从图片中提取垃圾类别、材质、来源和数量。

### 实现方式

SeeSea 采用三层识别架构：

1. 传统视觉规则
2. 本地 ONNX 检测器
3. 远程视觉大模型

### 融合逻辑

- 本地规则提供稳定底线
- ONNX 检测提供结构化候选框
- 视觉模型补充复杂场景理解、包装线索和来源判断

### 输出字段

- `item_name`
- `category`
- `material`
- `source_activity`
- `size_level`
- `quantity`
- `confidence`
- `evidence`

## 4.6 OCR 与包装特征匹配

### 目标

利用包装上的文字、图案和品牌线索增强来源判断能力。

### 实现

- OCR 提取包装文字
- 包装匹配字段写入识别结果
- 作为 evidence 的一部分参与结果解释

### 价值

- 支撑“垃圾身份证”概念
- 增强来源识别可信度

## 4.7 人工表单核对

### 目标

降低人工录入错漏，提高数据可信度。

### 实现

- 对 AI 识别条目与人工条目做数量和名称匹配
- 输出 AI 总数与人工总数
- 标记一致或差异
- 为管理员提供复核依据

### 价值

- 从“经验录入”转为“证据核对”
- 直接提升数据质量

## 4.8 志愿者反馈语义分析

### 目标

把自然语言反馈转成结构化风险信息。

### 实现

- 规则分析志愿者文本
- 提取密集度、复杂度、安全风险
- 生成简易风险结论

### 价值

- 丰富案例上下文
- 支持安全管理和任务优先级判断

## 4.9 扩散路径模拟

### 目标

从“垃圾是什么”进一步延伸到“可能从哪里来、会扩散到哪里”。

### 实现

- 基于潜点位置
- 基于来源类型
- 基于规则生成扩散点序列

### 输出

- 轨迹点列表
- 来源类型标签
- 概率值

## 4.10 LLM 报告系统

### 目标

将结构化识别结果转成面向项目管理、志愿者沟通和公众传播的自然语言报告。

### 实现

- 文本模型生成案例级报告
- 文本模型生成管理员指挥简报
- 文本模型生成潜水员汇总报告
- 当远程模型失败时自动降级到本地规则报告

### 技术细节

- SiliconFlow Chat Completions 接口
- JSON 结果解析
- 超时、重试、错误透传
- 连接测试接口

## 4.11 管理员总览与决策支持

### 实现

管理员端整合：

- 指标卡
- 图表
- 地图
- 风险预警
- AI 指挥简报
- 潜水员汇总

### 价值

这使 SeeSea 不再只是“识别工具”，而是“治理支持系统”。

## 4.12 公益成效与实施评估模块

### 目标

将技术结果自动翻译为评审关心的公益价值和实施价值。

### 实现

系统根据当前案例数据自动生成：

- 估算清理重量
- 自动化覆盖率
- 数据可信度
- 节省工时
- 部署模式
- 成本级别
- 运营负担
- 可持续行动建议

### 价值

- 直接服务答辩
- 让技术成果更易被公益导师理解
- 主要体现在技术报告、提交材料和 PPT 中，而非管理员首页展示模块

## 5. 可扩展性设计

## 5.1 模型接口可替换

系统已预留多种扩展入口：

- 视觉模型接口
- 文本模型接口
- 本地 ONNX 检测接口
- 本地规则接口

这意味着后续可无缝替换为：

- 更强的开源视觉模型
- 机构私有模型
- 新的 OCR 引擎
- 不同供应商的大模型 API

## 5.2 数据结构可扩展

每个案例已拆分为：

- 原始案例主表
- 人工表单子表
- 检测结果子表
- 语义分析子表
- 报告子表
- 扩散点子表

后续可以继续扩展：

- 视频片段
- 海流数据
- 设备信息
- 打捞重量
- 回访记录

## 5.3 部署模式可扩展

当前是单机轻量版，但可扩展为：

- 云数据库
- 异步任务队列
- 对象存储
- 多机构多项目 SaaS 版本

## 6. 创新性分析

SeeSea 的创新不在于单个模型本身，而在于把多种技术组织成公益机构真正能用的工作流：

### 创新 1：图像即数据

把水下照片从“展示素材”升级为“数据入口”。

### 创新 2：垃圾身份证

给每件垃圾补上类别、材质、来源、文字线索和证据来源。

### 创新 3：证据链式治理

从图像增强，到识别，到核对，到扩散模拟，到报告输出，形成完整证据链。

### 创新 4：公益友好型 AI

不是追求模型炫技，而是强调低成本、可部署、可维护、可降级。

它具备：
