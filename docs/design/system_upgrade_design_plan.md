# SeeSea 系统升级优化与总体设计规划

1. 从“图片识别”升级为“AI 证据链系统”：每个结论都记录来源、模型、版本、置信度、人工修正和时间。
2. 从“垃圾识别”升级为“垃圾知识图谱”：构建 Marine Debris Knowledge Graph，沉淀垃圾、来源、地点、人员、任务和治理行动之间的关系。
3. 从“规则漂流路线”升级为“多源环境驱动的漂移模拟”：接入海流、潮汐、风速风向、岸线地形等环境数据。
4. 从“LLM 报告生成”升级为“多智能体协作分析”：由视觉、溯源、安全、行动规划、影响评估等 Agent 共同输出治理策略。
5. 从“准确率”升级为“可信 AI 与人机协同”：管理员审核的目的不是简单改错，而是形成可信档案、训练样本和模型优化闭环。
6. 从“单点公益系统”升级为“区域治理数字底座”：支持多机构、多区域、多潜点、多批次治理行动长期运行。

## 3. 总体系统框架

### 3.1 端侧结构

系统保留两个用户端，但统一采用响应式 Web / PWA 架构，使手机移动端、网页端和主机端都能登录使用。

- 潜水员端：登录后选择“水下模式”或“岸上模式”。
- 后台管理员端：采用 CRM 风格的资源管理后台。
- 可选扩展端：小程序端、公众展示端、机构协作端。

### 3.2 服务分层

```text
用户交互层
  潜水员端（水下/岸上） | 管理员端 | 未来公众/机构端

业务编排层
  登录鉴权 | 日历任务 | 案例管理 | 审核流 | 任务派发 | 公益行动闭环

AI 决策层
  图像增强 | 多模型识别 | OCR | 证据链融合 | 风险分析 | 多 Agent 报告

空间智能层
  地图总览 | PostGIS | 轨迹建模 | OpenDrift | 不确定性预测 | 溯源沙盘

知识与数据层
  Marine Debris KG | 案例库 | 样本库 | 环境数据层 | 审核日志 | 影响评估
```

### 3.3 推荐技术演进

- 短期保留 FastAPI + SQLite，快速完成界面和业务闭环。
- 中期引入 Alembic 管理数据库迁移，避免手工改表。
- 中长期将 SQLite 升级为 PostgreSQL + PostGIS，用于空间查询、轨迹和区域治理。
- 图片文件从本地目录逐步升级为对象存储，保留本地演示降级模式。
- AI 任务从同步执行升级为后台任务队列，避免手机上传时长时间等待。
- 前端保持移动优先，可逐步升级为 PWA，支持离线暂存和弱网补传。

## 4. 端到端业务流程

1. 管理员创建潜水员账号、维护潜点、发布清理或巡查任务。
2. 潜水员登录后进入日历首页，查看历史工作、任务安排、累计工作时长和待处理事项。
3. 潜水员选择水下模式，完成签到、定位、拍照、备注、清理状态记录，并上传水下案例。
4. 系统为每张图片生成一个案例，保存原图、位置、时间、天气、海拔/深度、能见度、清理状态和上传人。
5. AI 管线执行图像增强、多模型识别、OCR、证据融合、风险初判和规则/环境漂移分析。
6. 潜水员上岸后进入岸上模式，对实物垃圾进行分类、称重、数量复核、补充志愿者心声和现场说明。
7. 系统将案例、垃圾条目、证据、称重、反馈、位置和任务关系写入 Marine Debris Knowledge Graph。
8. 管理员在后台审核案例，完成“人工复核 -> 修正入库 -> 样本沉淀 -> 模型优化”。
9. 对未清理或高风险垃圾，管理员派发任务给潜水员，形成任务执行和状态回传。
10. 多智能体报告系统输出单案例报告、区域溯源报告、行动计划和公益影响评估。
11. 清理完成后，系统更新清理状态、重量统计、影响指标和公众传播材料，形成公益行动闭环。

## 5. 潜水员端设计

### 5.1 潜水员首页

登录后先进入潜水员工作台，而不是直接进入上传表单。首页承担“任务入口 + 工作记录 + 模式选择”的作用。

核心功能：

- 模式选择：水下模式、岸上模式。
- 日历页：按日期查看历史案例、工作时间、任务安排和管理员留言。
- 任务列表：显示管理员分配的待执行任务、优先级、潜点、预计风险和截止时间。
- 工作时间汇总：按日、周、月统计潜水员工作时长、案例数、清理重量和任务完成量。
- 个人记录：查看本人历史上传、审核反馈和被退回修正的案例。

### 5.2 水下模式

水下模式面向“快速、少输入、弱网、现场安全”。

| 功能 | 对应需求 | 设计要点 |
| --- | --- | --- |
| 签到/开始作业 | 潜水员水下工作模式有签到功能 | 生成 DiveSession，记录开始时间、潜点、人员、任务 ID。 |
| 拍照上传案例 | 通过拍照上传案例 | 每张图片创建一个 Case，支持拍照、相册、批量补传。 |
| 自动定位 | 自动地图定位坐标 | 优先浏览器 GPS，失败时地图选点；保存经纬度、定位精度、来源。 |
| 海拔/深度记录 | 坐标 + 海拔 | 记录 altitude；如设备或外部表支持，扩展 depth、water_depth。 |
| 环境信息 | 是否清洁、天气等信息 | 保存天气、能见度、水流强度、清理状态、危险标记。 |
| 任务关联 | 查看管理员发送任务 | 上传时可绑定任务，形成任务执行证据。 |
| 弱网暂存 | 方便水下作业 | PWA 本地缓存，网络恢复后补传。 |
| 快速安全标记 | 风险分析基础 | 一键标记渔网缠绕、尖锐物、污染物、低能见度。 |

### 5.3 岸上模式

岸上模式面向“实物复核、称重、分类、补充叙事”。

| 功能 | 对应需求 | 设计要点 |
| --- | --- | --- |
| 案例复核列表 | 上岸后进行垃圾分类和称重 | 显示本次 DiveSession 下全部图片案例和 AI 初判。 |
| 垃圾分类录入 | 垃圾分类 | 支持按类别、材质、来源活动、数量、状态修正。 |
| 称重录入 | 文档要求增加称重录入模块 | 记录单件重量、批次重量、称重方式、单位、照片证据。 |
| AI 与人工差异 | 人工复核 | 显示 AI 识别和人工录入差异，提示漏项和数量偏差。 |
| 志愿者心声反馈 | 增加反馈模块 | 结构化保存志愿者文本、情绪、风险关键词和传播素材。 |
| 案例合并/拆分 | 每个图片为一个案例，但岸上有批次实物 | 支持多图对应同一批次垃圾，或一图拆出多个垃圾身份证。 |
| 清洁状态确认 | 是否清洁 | 确认 pending / in_progress / completed，并可上传清理后照片。 |

## 6. 管理员端设计

管理员端采用 CRM 风格，把“案例、垃圾、人员、任务、地图、报告、样本”都作为后台资源管理。

### 6.1 后台首页

核心信息：

- 大地图：展示潜点、垃圾案例、未清理点位、漂移轨迹和风险热区。
- 总体数据：案例数、垃圾件数、清理重量、未清理任务、高风险点、待审核数。
- 趋势统计：按时间、潜点、类别、材质、来源活动、潜水员维度统计。
- 运营提醒：高风险未处理、同源聚集、审核积压、任务逾期、模型低置信度。

### 6.2 潜水员管理

| 功能 | 对应需求 | 审核/管理价值 |
| --- | --- | --- |
| 潜水员列表 | 能看到每个潜水员 | 统一管理账号、组织、联系方式和状态。 |
| 活动轨迹 | 看到具体活动 | 关联 DiveSession、潜点、任务、上传案例。 |
| 工作内容 | 工作内容等信息 | 统计清理重量、案例数、工作时长、审核通过率。 |
| 绩效与安全 | 历史工作时间汇总 | 识别高贡献人员和高风险作业人员。 |
| 任务记录 | 管理员发送任务 | 查看任务接收、执行、完成和反馈过程。 |

### 6.3 垃圾管理

垃圾管理应从“案例列表”升级为“垃圾身份证列表”。

两类视图：

- 已清理垃圾：已确认清理完成，有称重、有清理照片或人工确认。
- 未清理垃圾：仍在水下或岸边待处理，可直接派发任务。

管理员操作：

- 查看单个垃圾身份证。
- 人工复核 AI 识别结果。
- 修正类别、材质、来源、数量、重量和风险等级。
- 对未清理垃圾创建任务并分配潜水员。
- 标记已清理、需复查、无法处理、疑似危险品。

审核目的必须明确：

- 纠正 AI 错误，保证治理台账可信。
- 将人工修正沉淀为训练样本。
- 标记高质量样本、困难样本、冲突样本。
- 为模型优化提供标签数据和评估集。
- 为公益传播和政府/机构报告提供可复核证据。

### 6.4 垃圾溯源与漂流分析

漂流溯源沙盘是项目特色，应从当前规则轨迹升级为“环境数据驱动的粒子漂移预测”。

输入：

- 单个垃圾案例：位置、时间、类别、材质、浮沉状态、重量、体积、图片证据。
- 区域聚合信息：同一海域、同一时间窗口、同一来源类型的垃圾分布。
- 环境数据接口：海流、潮汐、风速风向、浪高、海岸线、河口、港口、航道、渔区。

技术路线：

- 短期：保留规则模拟，增加置信区间和多情景轨迹。
- 中期：PostGIS 管理点、线、面和轨迹；用 Lagrangian particle tracking 表示粒子轨迹。
- 长期：接入 OpenDrift，使用真实海洋环境数据生成漂移路径和反向溯源概率。

输出：

- 可能来源区域。
- 未来扩散路径。
- 不确定性区间。
- 高风险搁浅或聚集区域。
- 对应行动建议：巡查、清理、拦截、宣传或上报。

### 6.5 报告分析生成

SeeSea 的 LLM 不只是写报告，而是承担多角色协同推理，把图像识别结果转化为治理策略。

推荐 Agent 体系：

| Agent | 输入 | 输出 |
| --- | --- | --- |
| 总控调度 Agent | 案例、图谱、轨迹、任务状态 | 拆解分析任务，汇总最终结论。 |
| 视觉分析 Agent | 原图、增强图、检测框、OCR | 垃圾类别、材质、证据解释、低置信提醒。 |
| 证据链审计 Agent | 多模型结果、人工修正、日志 | 判断证据是否充分，给出可信度和复核建议。 |
| 溯源推理 Agent | 图谱、区域分布、漂流结果 | 来源假设、扩散方向、相似案例。 |
| 安全评估 Agent | 垃圾类型、环境、反馈 | 潜水风险、处理风险、优先级。 |
| 行动规划 Agent | 未清理垃圾、人员、任务、天气 | 分配建议、路线、物资和复查计划。 |
| 公益影响 Agent | 重量、数量、志愿者反馈 | 影响指标、传播文案、项目复盘。 |

所有 Agent 输出都应结构化保存，并带来源引用，避免只生成一段不可复核的自然语言。

### 6.6 人员管理

人员管理覆盖账号和组织资源：

- 创建潜水员账号。
- 停用/恢复账号。
- 维护组织、手机号、证书、紧急联系人。
- 绑定角色：管理员、潜水员、审核员、机构观察员。
- 记录权限：案例查看范围、任务分配权限、报告导出权限。

### 6.7 区域治理扩展

未来 SeeSea 可从单机构工具升级为区域海洋垃圾治理数字底座：

- 多机构协作：公益组织、潜店、高校团队、治理部门。
- 多海域管理：按城市、海湾、保护区、潜点分层。
- 标准化数据接口：对接海洋环境数据、政府治理台账、公开科普平台。
- 长期趋势：年度清理量、重点来源变化、政策干预效果。
- 公众参与：发布清理成果、志愿者故事、认领任务和科普页面。

## 7. 垃圾身份证设计

每件垃圾不是一个普通识别框，而是一条结构化档案。

推荐字段：

- `debris_uid`：垃圾唯一编号。
- `case_id`：所属图片案例。
- `session_id`：所属潜水作业批次。
- `task_id`：关联任务。
- `category`：垃圾类别。
- `item_name`：具体名称。
- `material`：材质。
- `quantity`：数量。
- `weight_kg`：称重。
- `size_level`：尺寸等级。
- `bbox`：图像检测框。
- `image_refs`：原图、增强图、标注图、清理后照片。
- `geo`：经纬度、海拔、深度、定位精度。
- `environment`：天气、能见度、水流、潮汐、风向。
- `source_activity`：疑似来源活动。
- `risk_level`：风险等级。
- `cleanup_status`：未清理、处理中、已清理、需复查。
- `review_status`：待审核、已审核、退回、关闭。
- `evidence_chain`：AI、OCR、人工、漂移、图谱等证据。
- `kg_node_id`：知识图谱节点 ID。

## 8. AI 证据链设计

证据链的核心要求是：每个判断都给出来源。

建议为每条结论保存以下信息：

- 证据类型：视觉模型、YOLO、本地规则、OCR、人工表单、称重、志愿者反馈、漂流模拟、知识图谱。
- 来源名称：例如 YOLOv8、FUnIE-GAN、SiliconFlow GLM、OCR 引擎、管理员人工复核。
- 模型版本：模型名、权重版本、Prompt 版本。
- 输入对象：原图、增强图、检测框、文本、环境数据。
- 输出结论：类别、材质、来源、风险、轨迹、建议。
- 置信度：单模型置信度和融合后置信度。
- 证据解释：为什么给出这个判断。
- 人工处理：是否被管理员确认、修正或驳回。
- 时间戳：生成和审核时间。

融合逻辑：

1. YOLO 给出检测框和基础类别。
2. 图像增强模型提升输入质量，并记录增强前后质量指标。
3. OCR 提供包装文字、品牌、产地或使用场景线索。
4. VLM 提供复杂场景识别和来源活动推断。
5. 规则引擎提供可解释兜底。
6. 人工称重和分类作为高优先级事实。
7. 管理员审核结果写回样本库，形成模型优化闭环。

## 9. Marine Debris Knowledge Graph

### 9.1 节点

- Debris：单件垃圾。
- Case：图片案例。
- DiveSession：一次潜水作业。
- Diver：潜水员。
- Task：任务。
- Location：潜点、区域、岸线、河口、港口。
- Material：塑料、金属、玻璃、尼龙、复合材料。
- Category：包装废弃物、渔业废弃物、生活垃圾、大件漂浮垃圾。
- SourceActivity：旅游、渔业、航运、岸上活动、河流输入。
- Evidence：AI、OCR、人工、称重、漂流模拟。
- DriftRun：一次漂流模拟。
- Risk：缠绕、尖锐、污染、生态影响。
- CleanupAction：清理行动。
- Report：案例报告、区域报告、公益影响报告。

### 9.2 边

- Debris `observed_in` Case。
- Case `captured_by` Diver。
- Case `located_at` Location。
- Debris `made_of` Material。
- Debris `belongs_to` Category。
- Debris `likely_from` SourceActivity。
- Debris `supported_by` Evidence。
- Debris `simulated_by` DriftRun。
- Task `assigned_to` Diver。
- Task `targets` Debris 或 Location。
- CleanupAction `cleans` Debris。
- Report `summarizes` Case / Location / Task。

### 9.3 图谱价值

- 从单案例查询相似垃圾和历史来源。
- 从区域聚合判断同源污染。
- 将人工审核结果转为长期知识。
- 支撑 LLM Agent 做可引用的治理推理。
- 支撑未来跨机构、跨海域数据治理。

## 11. 分阶段实施路线

### 阶段一：产品结构升级

目标：先让双端流程完整贴合业务叙事。

- 潜水员端新增日历首页。
- 增加水下/岸上模式选择。
- 增加 DiveSession、任务、工时统计。
- 岸上端增加分类和称重录入。
- 管理员端增加未清理垃圾任务派发。

### 阶段二：垃圾身份证与证据链

目标：把当前 Detection 升级为 Debris 档案。

- 新增 DebrisItem 主体表。
- 新增 EvidenceRecord 证据表。
- 将 YOLO、VLM、OCR、规则、人工修正都写入证据链。
- 管理员审核时记录修正原因、审核人和样本标签。
- 报告中展示“结论来自哪里”。

### 阶段三：知识图谱与样本沉淀

目标：将案例数据沉淀为 Marine Debris Knowledge Graph。

- 定义图谱节点和边。
- 先用关系型表模拟图谱，后续可接入 Neo4j 或 RDF。
- 建立相似案例查询、同源预警、来源聚合。
- 将人工审核后的高质量样本导出给模型训练。

### 阶段四：漂流溯源沙盘

目标：把地图做成项目特色。

- 当前规则轨迹增加反向溯源和不确定性区域。
- 引入 PostGIS 存储轨迹和环境图层。
- 预留海流、潮汐、风速风向、岸线接口。
- 接入 OpenDrift 或同类粒子追踪框架。
- 管理员端提供单垃圾、区域聚合、时间窗口三种沙盘视图。

### 阶段五：多智能体治理报告

目标：让 LLM 从“写报告”升级为“协作决策”。

- 建立 Agent 输入输出 JSON Schema。
- 总控调度 Agent 调用视觉、证据、溯源、安全、规划和公益影响 Agent。
- 所有 Agent 输出带证据引用。
- 管理员可选择生成单案例、潜点、区域、月度行动报告。

### 阶段六：区域治理数字底座

目标：面向长期运行和多机构扩展。

- 多机构、多角色、多海域权限。
- 区域治理指标和年度趋势。
- 公众展示和公益传播接口。
- 标准化数据导出和外部系统 API。
- 模型评估、样本版本和治理效果评估。



这样能够清晰说明：SeeSea 不是一个“识别图片的小工具”，而是从现场采集、可信判断、溯源推理、行动规划到公益复盘的完整水下垃圾治理系统。
