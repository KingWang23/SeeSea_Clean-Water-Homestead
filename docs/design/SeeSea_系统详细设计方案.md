# SeeSea 水下垃圾治理多智能体 AI 决策系统详细设计方案

## 0. 文档目标

本文结合《需求文档.docx》《洁净水下家园潜水作业流程（简要说明附表100个海洋垃圾分类）.pdf》、现有 `system_upgrade_design_plan.md` 以及补充设计理念，给出 SeeSea 的完整系统设计。

设计目标不是做一个单纯的“水下图片识别工具”，而是构建从现场采集、垃圾识别、可信证据、漂流溯源、任务派发、公益影响评估到区域治理决策的闭环系统。

核心定位：

```text
SeeSea = 水下垃圾治理现场工具 + AI 证据链系统 + Marine Debris Knowledge Graph
       + 漂流溯源沙盘 + 多智能体治理报告 + 公益行动闭环平台
```

### 1.2 四个设计理念落地

| 设计理念 | 产品表达 | 技术表达 | 价值 |
| --- | --- | --- | --- |
| 垃圾身份证 | 单件垃圾详情页展示基础信息、位置、照片、AI 判断、人工修正、风险、来源和清理状态。 | `DebrisItem` 主实体 + `EvidenceRecord` + `KGNode` + `CleanupAction`。 | 让每件垃圾可追踪、可复核、可统计。 |
| AI 证据链 | 所有 AI 结论旁显示“来源、模型、置信度、输入和人工审核状态”。 | 模型注册表、证据表、融合引擎、审核日志。 | 从黑盒识别升级为可信 AI。 |
| 漂流溯源沙盘 | 后台大地图展示发现点、反向来源概率、未来扩散轨迹、不确定性范围和相似案例。 | PostGIS + Lagrangian particle tracking + OpenDrift 适配接口。 | 把地图从展示组件升级为治理决策核心。 |
| 公益行动闭环 | 未清理垃圾自动进入任务池，清理后统计重量、件数、志愿者反馈和传播素材。 | 任务状态机、影响评估指标、报告 Agent、公众展示接口。 | 从“看到问题”升级为“完成治理并证明影响”。 |

## 2. 总体业务闭环

### 2.1 端到端流程

```text
管理员建潜点/建账号/发任务
  -> 潜水员登录
  -> 日历工作台查看任务和历史
  -> 入水前签到、安全确认、选择水下模式
  -> 水下拍照/视频/定位/环境记录/清理状态
  -> 每张图片生成 Case，AI 后台异步处理
  -> 图像增强 -> YOLO 检测 -> OCR -> VLM 大模型识别 -> 规则校验 -> 证据融合
  -> 生成 DebrisItem 垃圾身份证和 EvidenceRecord 证据链
  -> 上岸后按本次作业批次分类、称重、数量复核、补充志愿者心声
  -> 写入 Marine Debris Knowledge Graph
  -> 管理员审核：人工复核 -> 修正入库 -> 样本沉淀 -> 模型优化
  -> 对未清理/高风险垃圾派发任务
  -> 漂流溯源沙盘做单件和区域分析
  -> 多 Agent 生成案例/区域/行动/公益影响报告
  -> 公益传播、复查任务、长期治理指标沉淀
```

### 2.2 关键状态机

#### DiveSession 作业批次状态

```text
draft 待开始
  -> checked_in 已签到
  -> underwater 进行中
  -> surfaced 已上岸
  -> shore_recording 岸上录入中
  -> submitted 已提交
  -> reviewed 已审核
  -> archived 已归档
```

需求对应：PDF-R2、PDF-R4、PDF-R5、PDF-R7、USER-P2、USER-P4。

#### Case 图片案例状态

```text
created 已创建
  -> uploaded 已上传
  -> ai_processing AI处理中
  -> ai_completed AI完成
  -> shore_confirmed 岸上确认
  -> admin_reviewing 管理员审核中
  -> approved 已入库
  -> returned 退回修正
  -> closed 关闭
```

需求对应：DOC-C1、DOC-S1、DOC-S2、USER-P5、USER-A4。

#### DebrisItem 垃圾身份证状态

```text
detected AI发现
  -> pending_cleanup 未清理
  -> assigned 已派发
  -> in_cleanup 清理中
  -> cleaned 已清理
  -> verified 已确认
  -> sample_archived 样本沉淀
```

需求对应：USER-D1、USER-A3、USER-D4。

#### Task 治理任务状态

```text
new 新建
  -> assigned 已分配
  -> accepted 潜水员已接收
  -> checked_in 已到场
  -> submitted 已提交结果
  -> verified 管理员确认
  -> reopened 复查
  -> closed 关闭
```

需求对应：USER-P3、USER-A2、USER-A3、USER-D4。

## 3. 系统用户与权限

### 3.1 用户角色

| 角色 | 使用端 | 主要目标 | 权限范围 | 对应需求 |
| --- | --- | --- | --- | --- |
| 潜水员/志愿者 | 潜水员端 | 执行水下采集、上岸分类称重、提交反馈。 | 查看自己的任务、作业批次、案例和审核反馈。 | PDF-R1、PDF-R3、PDF-R4、PDF-R5、USER-P2 |
| 潜点伙伴/领队 | 潜水员端 + 管理员轻权限 | 组织潜点活动、管理本潜点志愿者、初审数据。 | 查看本潜点数据、发起活动、汇总提交。 | PDF-R1、PDF-R7、DOC-C2 |
| 后台管理员 | 管理员端 | 管理人员、潜点、垃圾、任务、审核、报告。 | 全局资源管理、审核和导出。 | USER-A1 到 USER-A7 |
| AI 审核员 | 管理员端 | 复核模型结果、沉淀样本、维护分类体系。 | 案例审核、样本标注、模型评估。 | USER-A4、USER-D2 |
| 机构观察员 | 管理员端/未来机构端 | 查看区域治理数据和报告。 | 只读查看授权区域数据和报告。 | USER-A8、DOC-C2 |
| 公众用户 | 未来公众展示页 | 查看公益成果、传播数据和科普信息。 | 只读聚合数据，不查看敏感个人信息。 | DOC-S4、USER-D4 |

### 3.2 权限模型

采用 RBAC + 数据范围控制：

- RBAC：角色决定可访问模块，例如 `diver`、`site_partner`、`admin`、`reviewer`、`viewer`。
- 数据范围：按组织、潜点、城市、海域限制数据可见性。
- 操作审计：创建、审核、修正、导出、删除、任务派发均写入 `AuditLog`。
- 隐私保护：公众端不暴露潜水员手机号、精确活动轨迹、未审核照片和敏感风险点。

## 4. 多端产品形态

### 4.1 端侧结构

```text
响应式 Web / PWA
  ├─ 潜水员端
  │   ├─ 手机端：水下采集、弱网暂存、拍照、定位、岸上录入
  │   ├─ 平板端：批量分类、称重、复核
  │   └─ 桌面端：历史查看、补录、报告查看
  └─ 管理员端
      ├─ 手机端：待审核、任务提醒、地图轻量查看
      ├─ 桌面网页端：CRM 管理、地图沙盘、报告分析
      └─ 主机大屏端：沉浸式可视化、成果展示、区域看板
```

需求对应：DOC-C2、USER-P1。

### 4.2 技术选型建议

| 层级 | 演示版 | 生产演进 | 设计理由 |
| --- | --- | --- | --- |
| 前端 | Vue/React + 响应式布局 + PWA | Next.js/Nuxt + 组件库 + 地图 SDK | 同一套代码覆盖移动端、网页端、主机端。 |
| 地图 | Leaflet/Mapbox GL | Mapbox GL/Cesium/Deck.gl | 支持点位、热力、轨迹、粒子和沉浸式展示。 |
| 后端 | FastAPI | FastAPI + Celery/RQ + API Gateway | AI 流程适合异步任务和结构化接口。 |
| 数据库 | SQLite/PostgreSQL | PostgreSQL + PostGIS | 空间查询、轨迹、区域统计需要 PostGIS。 |
| 文件存储 | 本地目录 | MinIO/S3/OSS | 原图、增强图、标注图、称重照片、报告文件集中管理。 |
| 图谱 | 关系表模拟 | Neo4j/RDF/图数据库可选 | 前期降低复杂度，后期支持复杂关系推理。 |
| AI 服务 | Python service | 模型注册中心 + 推理队列 | YOLO、OCR、VLM、FUnIE-GAN、OpenDrift 独立升级。 |
| 报告 | LLM API | 多 Agent 编排 + 模板化报告 | 保证报告可复核、可导出、可审计。 |

### 5.3 水下模式

水下模式的 UI 原则是“少输入、大按钮、弱网可用、先安全后采集”。

#### 页面结构

```text
水下模式首页
  ├─ 当前作业卡片：潜点、领队、任务、已用时、同步状态
  ├─ 安全确认：标准潜水程序/免责协议/工具确认
  ├─ 签到按钮：开始作业
  ├─ 拍照上传：单拍/连拍/视频片段
  ├─ 快速标记：未清理/已清理/危险/需复查
  ├─ 环境记录：天气、能见度、水流、海况、海拔/深度
  └─ 结束作业：上岸，进入岸上模式
```

| `weather` | 天气 | USER-P4 |
| `visibility_level` | 水下能见度 | DOC-S1、DOC-S3 |

### 8.1 证据记录结构

```json
{
  "evidence_id": "EV-20260504-000001",
  "target_type": "DebrisItem",
```

### 8.2 证据类型

| 证据类型 | 示例来源 | 支持的判断 | 对应需求 |
| --- | --- | --- | --- |
| 图像增强证据 | FUnIE-GAN、OpenCV、UWCNN | 图像质量改善、可识别性 | DOC-S1 |
| 检测模型证据 | YOLO、Detectron、ONNX 模型 | 位置框、数量、基础类别 | DOC-S2、USER-P6 |
| VLM 证据 | SiliconFlow 大模型 API、其他视觉语言模型 | 复杂类别、材质、来源活动、风险解释 | DOC-C1、USER-P6 |
| OCR 证据 | PaddleOCR、Tesseract、云 OCR | 包装文字、品牌、产地、用途 | DOC-C1 |
| 表单证据 | 潜水员录入、岸上称重 | 重量、数量、清理状态、反馈 | PDF-R4、PDF-R5 |
| 人工审核证据 | 管理员复核 | 最终可信标签、修正原因 | USER-A4 |
| 漂流模拟证据 | OpenDrift、规则模型 | 来源区域、扩散路径、不确定性 | USER-A5 |
| 图谱证据 | 相似案例、区域聚合 | 同源判断、历史趋势 | USER-D1、USER-D3 |

### 8.3 多模型融合逻辑

```text
输入：原图 + 元数据 + 潜水员初始表单
  1. 图像质量评估：模糊、偏色、噪声、气泡遮挡
  2. 图像增强：生成增强图和质量指标
  3. YOLO 检测：输出 bbox、基础类别、数量
  4. OCR 识别：输出包装文字、品牌、数字、语言
  5. VLM 分析：输出类别、材质、来源活动、风险解释
  6. 分类体系映射：映射到 PDF 100 类 taxonomy_code
  7. 规则校验：材质-类别一致性、危险品规则、数量异常
  8. 证据融合：生成候选结论和置信度
  9. 人工复核：上岸端/管理员确认或修正
  10. 样本沉淀：审核通过数据进入训练样本库
```

### 8.4 模型接口抽象

所有模型通过统一接口接入，方便未来升级。

```python
class ModelAdapter:
    name: str
    version: str
    capability: list[str]

    def predict(self, inputs: dict) -> dict:
        ...
```

适配器类型：

- `EnhancementAdapter`：FUnIE-GAN、UWCNN、OpenCV white balance。
- `DetectionAdapter`：YOLO、Detectron、ONNX Runtime。
- `OCRAdapter`：PaddleOCR、Tesseract、云 OCR。
- `VLMAdapter`：SiliconFlow API、其他多模态大模型。
- `DriftAdapter`：规则漂移、OpenDrift、外部海洋模型。
- `LLMAgentAdapter`：报告生成和多 Agent 协作。

接口输出统一写入 `EvidenceRecord`，避免模型替换影响业务层。

| `Diver` | 潜水员/志愿者 | 组织、活动、贡献、审核质量 |
| `DiveSite` | 潜点 | 城市、海域、经纬度、历史趋势 |
| `Location` | 位置/区域 | 点、线、面、岸线、河口、港口 |
| `Material` | 材质 | 塑胶、金属、玻璃、橡胶等 |
| `Taxonomy` | PDF 100 类分类 | 编号、名称、材质 |
| `SourceActivity` | 疑似来源活动 | 渔业、旅游、航运、岸上生活、河流输入 |
| `Evidence` | 证据 | 来源、模型、置信度、解释 |
| `DriftRun` | 漂流模拟 | 输入、轨迹、置信区间、版本 |
| `Risk` | 风险 | 安全风险、生态风险、处理建议 |
| `Task` | 治理任务 | 类型、目标、人员、状态 |
| `CleanupAction` | 清理行动 | 清理前后、重量、结果、证据 |
| `Report` | 报告 | 类型、数据范围、Agent 输出 |

### 9.2 图谱关系

| 关系 | 含义 |
| --- | --- |
| `Debris observed_in Case` | 垃圾来自某个图片案例。 |
| `Case captured_by Diver` | 案例由潜水员拍摄。 |
| `Case belongs_to DiveSession` | 案例属于一次作业。 |
| `DiveSession located_at DiveSite` | 作业发生在潜点。 |
| `Debris made_of Material` | 垃圾材质。 |
| `Debris classified_as Taxonomy` | 垃圾分类。 |
| `Debris likely_from SourceActivity` | 可能来源活动。 |
| `Debris supported_by Evidence` | 判断由证据支撑。 |
| `Debris simulated_by DriftRun` | 漂流模拟关联。 |
| `Task targets Debris/Location` | 任务目标。 |
| `Task assigned_to Diver` | 任务分配给潜水员。 |
| `CleanupAction cleans Debris` | 行动清理了垃圾。 |
| `Report summarizes Case/DiveSite/Region` | 报告总结数据范围。 |

### 9.3 图谱查询场景

- 查询某件垃圾的完整证据链和人工修正历史。
- 查询同一潜点近 12 个月塑胶垃圾变化趋势。
- 查询某片海域渔业废弃物是否持续聚集。
- 查询“渔网 + 高风险 + 未清理”的任务池。
- 查询某类包装文字对应的可能来源活动。
- 查询某个潜水员参与的清理行动和贡献影响。
- 为 LLM Agent 提供可引用事实，而不是让模型自由编造。

## 10. 漂流溯源沙盘设计

### 10.1 产品形态

漂流溯源沙盘是后台地图的高级模式，也是 SeeSea 区别于普通垃圾识别系统的核心亮点。

页面结构：

```text
漂流溯源沙盘
  ├─ 地图主画布
  │   ├─ 垃圾发现点
  │   ├─ 可能来源区
  │   ├─ 未来扩散轨迹
  │   ├─ 不确定性包络
  │   ├─ 海流/潮汐/风场图层
  │   └─ 潜点/岸线/河口/港口/航道/渔区图层
  ├─ 左侧输入面板
  │   ├─ 单件垃圾 / 区域聚合 / 时间窗口
  │   ├─ 材质、浮沉状态、重量、体积
  │   ├─ 模拟时间范围
  │   └─ 环境数据源选择
  ├─ 右侧结果面板
  │   ├─ 来源概率排名
  │   ├─ 扩散路径摘要
  │   ├─ 高风险聚集区
  │   └─ 行动建议
  └─ 底部时间轴
```

### 10.2 分析模式

| 模式 | 输入 | 输出 | 对应需求 |
| --- | --- | --- | --- |
| 单件垃圾溯源 | 单个 `DebrisItem` 的位置、时间、材质、重量、浮沉状态 | 反向来源概率、可能污染源、证据说明 | DOC-C1、USER-A5 |
| 区域聚合溯源 | 同一潜点/海域/时间窗口的垃圾集合 | 同源聚集判断、来源活动占比、治理建议 | USER-A5、USER-D3 |
| 未来扩散预测 | 当前垃圾点和环境数据 | 未来路径、搁浅概率、扩散范围 | DOC-C1、USER-A5 |
| 清理行动规划 | 未清理点位、潜水员、天气、风险 | 路线、优先级、任务建议 | USER-D4、USER-A6 |

### 10.3 技术路线

#### 阶段一：规则模拟

- 根据发现点、材质、风向、水流方向生成简化轨迹。
- 生成多条情景线：保守、中性、激进。
- 用圆形/扇形表达不确定性。
- 适合原型演示，依赖数据少。

#### 阶段二：PostGIS 时空轨迹

- 使用 `geometry(Point, 4326)` 存储发现点。
- 使用 `geometry(LineString, 4326)` 存储轨迹。
- 使用 `geometry(Polygon, 4326)` 存储来源概率区和不确定性包络。
- 支持按区域、时间、距离、热区进行空间查询。

#### 阶段三：OpenDrift 粒子漂移

- 使用 Lagrangian particle tracking。
- 针对每件或每组垃圾释放粒子。
- 输入海流、潮汐、风速风向、浪高、岸线地形。
- 输出粒子轨迹集合、概率密度、搁浅点和置信区间。

#### 阶段四：环境数据接口

预留接口：

- 海流数据。
- 潮汐数据。
- 风速风向。
- 浪高。
- 海岸线地形。
- 河口、排污口、港口、码头、航道、渔区。
- 天气历史和预报。

## 11. 多智能体报告与决策系统

### 11.1 Agent 体系

SeeSea 的 LLM 不只是写报告，而是把视觉识别、证据审计、溯源推理、安全评估和行动规划组织成可复核的推理链。

| Agent | 输入 | 输出 | 对应需求 |
| --- | --- | --- | --- |
| 总控调度 Agent | 用户选择的数据范围、案例、图谱、漂流结果 | 分析任务拆解、调用顺序、最终报告结构 | USER-A6 |
| 视觉分析 Agent | 原图、增强图、检测框、OCR、模型结果 | 类别、材质、数量、低置信提醒、视觉依据 | DOC-C1、DOC-S2 |
| 图像质量 Agent | 原图、增强图、质量指标 | 是否可用于识别、是否需要人工复核 | DOC-S1 |
| 证据链审计 Agent | 全部 `EvidenceRecord`、人工修正 | 证据充分性、冲突点、可信度 | USER-D2、USER-A4 |
| 溯源推理 Agent | 漂流运行、图谱、区域分布、OCR 文本 | 来源假设、污染源概率、相似案例 | DOC-C1、USER-A5 |
| 安全评估 Agent | 垃圾类别、材质、环境、反馈 | 潜水风险、处理风险、优先级 | DOC-S3、USER-P6 |
| 行动规划 Agent | 未清理垃圾、人员、天气、任务状态 | 任务分配建议、路线、物资、复查计划 | USER-D4、USER-A3 |
| 公益影响 Agent | 重量、数量、志愿者反馈、历史趋势 | 影响指标、传播摘要、项目复盘 | USER-D4、DOC-S4 |

### 11.2 Agent 输出规范

所有 Agent 输出必须结构化，并引用证据 ID。

```json
{
  "agent_name": "source_reasoning_agent",
  "conclusion": "该渔网碎片较可能来源于附近渔业活动。",
  "confidence": 0.72,
  "evidence_refs": ["EV-001", "DRIFT-001", "KG-QUERY-001"],
  "reasoning_summary": [
    "视觉证据显示网格状尼龙结构。",
    "同区域 30 天内存在多件渔具类垃圾。",
    "反向漂流结果指向东北侧渔区概率较高。"
  ],
  "risks": ["漂流模型暂未接入实测海流，置信度受限。"],
  "recommended_actions": ["对东北侧渔区边缘开展复查巡查。"]
}
```

### 11.3 报告生成流程

```text
管理员选择报告范围
  -> 总控调度 Agent 读取图谱和数据库
  -> 视觉分析 Agent 汇总识别证据
  -> 证据链审计 Agent 检查冲突和低置信结论
  -> 溯源推理 Agent 结合漂流沙盘和区域聚合
  -> 安全评估 Agent 生成风险等级和注意事项
  -> 行动规划 Agent 形成任务建议
  -> 公益影响 Agent 输出成果指标和传播摘要
  -> 总控调度 Agent 汇总为报告草稿
  -> 管理员人工确认
  -> 导出 PDF/Word/网页，并归档到 Report
```

需求对应：DOC-C1、DOC-C2、USER-A6、USER-D2、USER-D4。

## 13. 图像增强、多模型识别与校验实现

### 13.1 图像增强

输入：

- 原图或视频帧。
- 拍摄设备信息。
- 水下场景元数据：能见度、水深、光照、气泡。

处理：

- 白平衡和颜色校正。
- 去噪。
- 去雾/去散射。
- 锐化。
- FUnIE-GAN 或同类模型增强。

输出：

- 增强图。
- 图像质量评分。
- 增强模型版本。
- 是否建议人工复核。

需求对应：DOC-S1、USER-P6。

### 13.2 YOLO 检测

目标：

- 检测垃圾位置框。
- 估计数量。
- 识别基础类别。

处理逻辑：

- 对原图和增强图分别检测。
- 如果增强图显著提升置信度，优先采用增强图结果。
- 如果原图和增强图冲突，标记为证据冲突，进入人工复核队列。

需求对应：DOC-S2、USER-P6。

### 13.3 OCR 文字识别

目标：

- 识别包装文字、品牌、容量、生产信息、用途词。
- 为来源活动判断提供证据。

示例：

- “矿泉水”“饮料” -> 生活/旅游活动线索。
- “机油”“润滑油” -> 船舶/维修活动线索。
- “鱼饵”“钓具” -> 渔业/垂钓活动线索。

需求对应：DOC-C1、USER-P6。

### 13.4 VLM 大模型识别

使用 SiliconFlow 大模型 API 或其他多模态模型，作为复杂场景理解和可解释判断层。

输入：

- 原图。
- 增强图。
- YOLO bbox 裁剪图。
- OCR 文本。
- 潜点、时间、环境元数据。
- PDF 100 类分类体系。

输出：

- 候选分类 Top N。
- 材质。
- 来源活动。
- 风险点。
- 判断依据。
- 是否需要人工复核。

需求对应：DOC-C1、DOC-S2、USER-P6、USER-D2。

### 13.5 表单智能校验

校验规则：

- AI 检测数量与潜水员录入数量差异过大，提示复核。
- 分类为“渔网/渔线”但未标记潜水安全风险，提示补充。
- 重量为 0 或缺失，但状态为已清理，提示补称重。
- 填写“已清理”但没有清理后照片，提示补证据。
- 5 日内未提交完整数据，提醒潜点伙伴。

需求对应：DOC-S2、PDF-R4、PDF-R5、PDF-R7、USER-D2。

## 15. 公益行动闭环设计

### 15.1 闭环链路

```text
发现垃圾
  -> 生成垃圾身份证
  -> AI 证据链判断类别、来源、风险
  -> 未清理/高风险进入任务池
  -> 管理员派发任务
  -> 潜水员执行并上传清理证据
  -> 岸上分类称重
  -> 管理员审核
  -> 公益影响评估
  -> 生成报告和传播素材
  -> 区域治理策略更新
```

### 15.2 公益影响指标

| 指标 | 计算方式 | 对应价值 |
| --- | --- | --- |
| 清理件数 | `DebrisItem` 已清理数量 | 行动规模 |
| 清理重量 | `WeighingRecord.weight_kg` 汇总 | 治理成果 |
| 参与人次 | `DiveSession` 参与人员统计 | 志愿者动员 |
| 潜点覆盖 | 有作业记录的潜点数 | 区域覆盖 |

## 18. 分阶段建设路线

### 阶段一：业务闭环 MVP

目标：先让系统完整跑通现场流程。

- 完成潜水员端登录、日历、水下/岸上模式。
- 完成签到、拍照上传、定位、天气/环境、清理状态。
- 完成岸上分类、称重、数量录入、志愿者反馈。
- 完成管理员首页、潜水员管理、垃圾管理、任务派发。
- 建立 `Case`、`DebrisItem`、`WeighingRecord`、`Task` 核心表。

### 阶段二：垃圾身份证与 AI 证据链

目标：把识别结果升级为可信结构化档案。

- 接入图像增强、YOLO、OCR、SiliconFlow VLM。
- 建立 `EvidenceRecord`。
- 建立模型注册和适配器接口。
- 完成多模型融合和人工复核页面。
- 审核通过数据进入样本库。

### 阶段三：知识图谱和后台 CRM 深化

目标：让案例、垃圾、人员、任务、潜点、来源可关联查询。

- 建立 `kg_nodes`、`kg_edges`。
- 实现垃圾身份证详情页的图谱关系。
- 实现相似案例查询和同源聚集预警。
- 完成审核质量、模型错误类型统计。

### 阶段四：漂流溯源沙盘

目标：把地图做成项目特色。

- 地图支持点位、热力、轨迹、年份和潜点筛选。
- 引入 PostGIS 管理点线面。
- 完成规则漂流和不确定性可视化。
- 预留海流、潮汐、风、岸线接口。
- 接入 OpenDrift 进行粒子漂移预测。

### 阶段五：多智能体报告与公益影响

目标：让 LLM 承担治理决策推理。

- 建立 Agent 输入输出 JSON Schema。
- 实现总控调度、视觉、证据审计、溯源、安全、行动规划、公益影响 Agent。
- 生成单件、单次作业、潜点、区域和公益影响报告。
- 报告带证据引用，并需管理员确认后发布。

### 阶段六：区域治理数字底座

目标：从单机构工具升级为区域治理基础设施。

- 多机构、多区域、多潜点权限。
- 标准化数据接口和导出。
- 年度趋势和政策干预效果评估。
- 公众展示页和大屏沉浸式成果展示。
- 样本版本、模型版本和治理效果长期评估。

## 19. 验收标准

### 19.1 产品验收

- 潜水员可在手机端完成签到、拍照、定位、上传和岸上称重分类。
- 管理员可在后台查看地图、潜水员、垃圾身份证和任务。
- 未清理垃圾可被派发为任务，并完成回传和审核。
- 每件垃圾都有分类、照片、位置、重量、状态和证据链。
- 报告可按单件、潜点、区域生成，并包含证据引用。

### 19.2 AI 验收

- 图像增强前后均保存，模型版本可追溯。
- YOLO、OCR、VLM 结果均写入证据表。
- AI 与人工冲突时可提示复核。
- 人工修正能沉淀为训练样本。
- 报告中的 AI 结论能定位到证据 ID。

### 19.3 数据验收

- PDF 100 类分类作为标准 taxonomy 初始化。
- 每张图片对应一个 `Case`。
- 每个识别对象对应一个或多个 `DebrisItem`。
- 称重、分类、反馈、清理状态均可追溯到作业批次。
- 地图数据支持按时间、潜点、类别、状态过滤。

### 19.4 治理闭环验收

- 从发现未清理垃圾到任务派发、执行、审核、归档完整闭环。
- 可统计清理件数、重量、参与人次、任务闭环率。
- 可生成区域溯源分析和行动建议。
- 可输出公益传播素材，但敏感信息可控。

## 20. 总结

SeeSea 的系统框架应围绕四个核心能力展开：

1. 用“垃圾身份证”把单次识别结果变成长期治理资产。
2. 用“AI 证据链”让识别、来源、风险和报告结论可解释、可审核、可优化。
3. 用“漂流溯源沙盘”把地图从成果展示升级为污染源推理和行动规划工具。
